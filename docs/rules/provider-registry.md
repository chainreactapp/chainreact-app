# Rule: Provider Registry

## Purpose

Define how ChainReactV2 enumerates, looks up, and exposes capabilities for every supported integration provider. The registry is assembled from per-provider manifest files at module load time; no central god-object holds provider definitions.

## Resolved Decisions

**Locked for Slice 1:**
- Each provider has `integrations/<provider>/manifest.ts` exporting a `ProviderManifest`. The manifest IS the registry entry — no separate provider definition.
- `integrations/_registry.ts` aggregates manifests at module load time, **hand-maintained** (explicit imports per provider), not auto-generated by filesystem scan.
- Manifest contract lives in `contracts/integration.ts`. Build-time validation of every manifest against the contract.
- UI metadata (icon path, display order, marketing copy) lives in a separate `integrations/<provider>/ui.ts` colocated with the provider, not in the manifest.
- Provider id naming preserved from V1 (`slack`, `gmail`, `discord`, etc.) — backward compat for token rows during migration.
- Health-check intervals per provider declared in the manifest (Google/Microsoft 6h, Slack/Discord/GitHub 4h, others 12h).
- Capability checks always go through `providerSupports(id, capability)` — no string comparison anti-patterns.

**Deferred decisions:**
- Per-environment OAuth client ID overrides: manifest declares env-var names; env values supplied at runtime.
- Capability versioning when a future provider adds a new capability not in the contract.

**Decisions requiring product-owner input:**
- For `isEnabled: false` providers with active integrations: deprecation timeline + user notification copy.

## Current V1 problem being solved

V1's `lib/integrations/provider-registry.ts` is 1,595 lines and centrally declares every provider's capabilities, scopes, callback configuration, and data fetchers. Adding or modifying a provider means editing this file. Provider-specific changes risk unrelated providers.

V1 also mixes the registry's content with multiple secondary sources: `oauthConfig.ts` (662 lines), `availableIntegrations.ts` (603 lines), `integrationScopes.ts`, and `scope-validator.ts`. Different parts of the codebase consult different sources for the same fact ("does Slack support webhooks?"), and they don't always agree.

## V2 intended behavior

Each provider has a single `integrations/<provider>/manifest.ts` exporting a `ProviderManifest`. The manifest is the registry entry. There is no separate provider definition.

`integrations/_registry.ts` aggregates manifests at module load time:

```
import { slack } from './slack/manifest'
import { gmail } from './gmail/manifest'
// ...
export const PROVIDERS = Object.freeze({ slack, gmail, /*...*/ })
export function getProvider(id: ProviderId): ProviderManifest | undefined { ... }
export function listProviders(): readonly ProviderManifest[] { ... }
export function providerSupports(id: ProviderId, capability: Capability): boolean { ... }
```

Lookup is by stable provider id (`slack`, `gmail`, etc.). The registry exposes typed capability checks; callers do not destructure manifests.

## Single source of truth

- Per-provider definition: `integrations/<provider>/manifest.ts`.
- Aggregated registry: `integrations/_registry.ts` — a derived projection of all manifests.
- Manifest contract: `contracts/integration.ts` (the `ProviderManifest` Zod schema).

The registry is read-only at runtime. Manifests are statically importable so the registry can be analyzed at build time (TypeScript type-narrowing on `ProviderId`, build-time validation against the contract).

## Allowed flows

- **Adding a provider:** create `integrations/<provider>/` folder with `manifest.ts`, `oauth.ts`, `client.ts`, `webhooks/`, `triggers/`, `actions/`. The registry auto-includes once the manifest is imported in `_registry.ts`.
- **Reading capabilities:** any service that needs to know "does provider X support webhooks" calls `providerSupports(id, 'webhook-trigger')`. No local capability tables.
- **Listing providers:** UI reads `listProviders()` for the integrations page. Filtering by `isEnabled` happens in the UI layer.
- **Schema validation:** at module load, every manifest is validated against the `ProviderManifest` Zod schema. A malformed manifest fails the build.

## Disallowed behavior

- Provider definitions outside their `manifest.ts`. No sidecar files, no inline objects in services.
- Capability claims derived from anywhere other than the manifest (no "if provider id starts with ... then it supports webhooks").
- Mutation of the registry at runtime. Frozen object.
- A single manifest knowing about another provider. Each manifest is independent.
- Capability checks via string comparison (`providerId === 'slack'`). Always go through `providerSupports()` so capability is data, not branching.
- Manifests containing callable logic (oauth handlers, action handlers). Manifest is data only.

## Edge cases

- **Provider feature flags:** experimental providers gated behind an env var. Manifest declares `isExperimental: true`; registry lookup returns the manifest but UI hides it unless flag is set.
- **Versioned provider APIs:** Microsoft Graph v1 vs beta. Manifest declares `apiVersion: 'v1.0'`. Action/trigger schemas inside the same provider folder reference the manifest's version.
- **Provider with multiple OAuth app types (Slack: bot vs user tokens):** manifest declares `oauthFlows: ['bot', 'user']`; per-flow scopes nested. OAuth dispatcher reads the requested flow.
- **Disabled provider:** manifest stays in source but exports `isEnabled: false`. Registry lookup still returns the manifest (so existing tokens don't crash on read), but new connect flows refuse with a humanized message.
- **Provider rename / id change:** breaks integration row foreign keys and webhook registrations. Don't rename. If a provider rebrands, keep the old id and add a `displayName` override in the manifest.
- **Workspace-scoped vs user-scoped tokens:** manifest declares `tokenScope: 'workspace' | 'user'`. Workspace-scoped providers (Slack) require the discriminator field name (`team_id`); user-scoped providers (Gmail) do not.
- **Provider with no public webhooks:** `capabilities.webhookTrigger: false`. Triggers from this provider must use polling (declared in the trigger schema, not the manifest).

## Required tests

Unit tests in `tests/unit/integrations/_registry.test.ts`:

1. `_registry.ts` exports a frozen object.
2. `getProvider(knownId)` returns the manifest.
3. `getProvider(unknownId)` returns undefined.
4. `listProviders()` returns all manifests including disabled.
5. Every manifest passes `ProviderManifest` Zod validation at load.
6. `providerSupports('slack', 'webhook-trigger')` returns true (or whatever Slack actually supports).
7. `providerSupports('unknown', anyCapability)` returns false (does not throw).
8. `isExperimental` providers are excluded from default list when feature flag off.

Per-provider manifest tests at `tests/unit/integrations/<p>/manifest.test.ts`:

9. Each manifest declares all required fields per the contract.
10. Each manifest's `id` matches its folder name.
11. `scopes.required` is non-empty for OAuth providers.
12. `apiVersion` is declared if the provider has versioned APIs.

Build-time check (CI):

13. Adding a new provider folder without registering its manifest in `_registry.ts` is detected and fails CI.

## V1 behavior to preserve

- Provider id naming (`slack`, `gmail`, `discord`, etc.) — backward compat for token rows during migration.
- Capability semantics (what "supports webhooks" means).
- Distinction between OAuth provider and API key provider.
- Health-check intervals per provider class (Google/Microsoft 6h, Slack/Discord/GitHub 4h, etc.) — declared in manifest.

## V1 behavior to drop

- The 1,595-line god-object file.
- The split across `provider-registry.ts`, `oauthConfig.ts`, `availableIntegrations.ts`, `integrationScopes.ts`, `scope-validator.ts`. All collapse into per-provider manifests + the small `_registry.ts` aggregator.
- Capability-by-string-comparison anti-patterns scattered through V1 services.

## Open questions

(UI metadata, manifest discovery, and per-environment overrides previously listed here are now resolved — see "Resolved Decisions" above.)

1. **Capability versioning:** if a future provider adds a new capability not in the contract, the contract evolves. Manifest must declare unknown capabilities as `false` by default. Recommendation: capability list lives in `contracts/integration.ts` as a typed union. Deferred — concrete approach finalized at the time the first new capability lands.
2. **`isEnabled: false` for a provider with active integrations:** what does the UI show, and what happens to existing tokens? Recommendation: flag as "deprecated"; existing integrations keep working but new connections refused; users see a one-time notice. Requires product-owner input on deprecation timeline + notification copy (already flagged in Resolved Decisions).
